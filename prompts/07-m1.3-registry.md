# Prompt 07 — M1.3: Component metadata registry (JSON Schema)

> Use with `00-orchestrator.md` (v2) as system prompt. **Effort `xhigh`.** Run **after M1.2** (it queries through the wrapper).
> Read first: `metrocalk.md` (describe-to-create + provides/requires is the product's brain), `architecture.md` (registry row + invariants), ADR-004 (the registry later feeds search + the marketplace; same data path).

---

<task>
Build the component metadata registry — the data layer behind the compatibility query and, later, describe-to-create. Components register their field schema and their relational capabilities; the registry is what makes "what can this bind to?" and "what provides Health?" answerable, and is the same data the browser backend (ADR-006) and the marketplace (ADR-004) will read.
</task>

<scope>
In scope: runtime component registration from JSON Schema (fields + types); provides/requires/observes capability declarations; tags + aliases (the describe-to-create search surface, even though search itself is later); validation at registration (malformed schema rejected loudly, not panicking); serialize/round-trip. Wire registration so capabilities are queryable through the M1.2 wrapper.
Out of scope: describe-to-create search/LLM, embeddings, the marketplace, UI inspector generation (all consume this later), the commit pipeline. Build the registry + its query hooks only.
</scope>

<deliverables>
1. Registry supporting: register a component kind from a JSON Schema (fields/types); declare provides/requires/observes capabilities; tags + aliases; optional UI hints. Registration is runtime (not compile-time-only), so plugins/marketplace can register later.
2. Validation: a malformed schema is rejected at registration with an actionable error message; no panic, no silent acceptance.
3. Integration with the M1.2 wrapper: "what provides Health?" returns the correct kinds via the wrapper's pair-match query — the registry feeds capabilities into the ECS as relationship pairs.
4. **≥10 real example components** (Health, HealthBar, Transform, Sprite, RigidBody, Collider, …) registered via a builder or derive — whichever is less magic — and each queried + round-trip serialized.
</deliverables>

<success_criteria>
register N kinds from JSON Schema · "what provides Health?" returns them through the wrapper · malformed schema → clear, actionable error (tested) · all ≥10 example components register, query, and round-trip serialize identically · tags/aliases stored and retrievable, ready for later search.
</success_criteria>

<verification>
Round-trip test: register → serialize → deserialize → byte-identical metadata. Feed a deliberately malformed schema and assert the error is actionable (not a generic panic). Re-read against ADR-004: confirm a marketplace component would register through this same path; note any gap that Phase 2 must close.
</verification>
