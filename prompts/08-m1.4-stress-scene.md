<task>
Promote the spikes' throwaway scene generators into one shared, seeded, deterministic stress-scene generator used by both tests and benches — and use it to resolve spike ②'s open **F1** risk: does marking capability relationships `DontFragment`/sparse (or storing capability sets as data) collapse the ~14.8 KB/entity archetype-fragmentation overhead without hurting query latency?
</task>

<scope>
In scope: a shared scene generator (`/tools/scene-gen` or a `/core` test/bench util) — seeded with the same SplitMix64 seed as the spikes, parameterized (entity count, component distribution, binding density), with the 5k and 20k scenes as named presets; the F1 re-measurement through the M1.2 wrapper.
Out of scope: the CI perf gate (M1.5 wires it), the commit pipeline, any new query features. Shared fixture + the F1 measurement only.
</scope>

<deliverables>
1. **Shared seeded generator**: byte-identical scenes across runs and OS (verify by digest); 5k + 20k presets; usable by BOTH `cargo test` fixtures and criterion benches without duplication.
2. **F1 re-measurement**: capability relationships marked `DontFragment`/sparse (or modeled as capability-set data), with bytes/entity recorded against the **14.8 KB/entity** pair-model baseline, AND the cached compat-query p99 re-checked to confirm the storage change didn't regress latency.
3. A short recorded verdict: which storage model M1 adopts, with the measured memory AND latency for each candidate — so M1–2 builds on a decided model.
</deliverables>

<success_criteria>
byte-identical scene across runs (digest match) · bytes/entity WITH DontFragment recorded and **< 14.8 KB/entity** baseline · cached compat query still **<16 ms p99** under the new storage model · the generator is the single shared source for tests and benches.
</success_criteria>

<verification>
Two runs per discipline; report median + p99; investigate only >25% run-to-run swings. Confirm determinism by diffing scene digests across runs/OS. If `DontFragment` does NOT beat the baseline, say so plainly and record the fallback (capability-sets-as-data) with its numbers — do not force a win.
</verification>
