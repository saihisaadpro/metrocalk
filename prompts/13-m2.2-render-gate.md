# Prompt 13 — M2.2: Real-scene render gate (wgpu, native + browser)

> Use with `00-orchestrator.md` (v2). **Effort `xhigh`** — benchmark discipline applies to every number. Resolves the architecture open-question "real-scene render cost at ≥5k entities" (M0 only proved a *triangle*).
> **Lane:** Owns a throwaway `/spikes/render-scene` (+ wasm build) · Blocked by nothing (wgpu; independent of `/core` and the Tauri shell) · **Parallel-safe with M2.1 (12) and M1.6 (11)** · see `prompts/README.md`.
> Read first: ADR-003 (non-bindless web path), spike ③ (the wgpu/wasm baseline), and the **research findings below**.

---

<task>
Measure the real cost of rendering the editor stress scene (not a triangle) through wgpu — native AND in the browser — and confirm the rendering approach holds the frame budget at scale. This is the second M2 risk, independent of the Tauri shell.
**Research findings (verified June 2026):** instancing is mandatory (on the web, WebGPU per-draw overhead is *worse* than WebGL — non-instanced thousands-of-meshes scenes drop ~4×); reduce draw calls via instancing + batching + **GPU frustum culling (compute → indirect draws)** + render bundles; browser caveats — **no multi-draw-indirect**, `indirect-first-instance` is optional, largest storage buffer must be **< 128 MB**; watch for the wgpu periodic-slow-frame pattern; confirm the Windows DXGI frame-pacing fix is in your pinned wgpu.
</task>

<scope>
In scope (`/spikes/render-scene`): instanced rendering of the M1.4 stress scene at **5k + 20k** entities (entity meshes + instanced gizmos + grid) with a moving camera + frustum culling; native + wasm32/WebGPU; measure GPU frame time and CPU submit **separately**. Out of scope: the editor, picking/gizmo interaction logic, real art assets (use placeholder instanced meshes), the Tauri shell.
</scope>

<deliverables>
1. Instanced render of the 5k + 20k scene (reuse M1.4's `scene-gen` for entity transforms); instanced gizmos; **GPU frustum culling via compute → indirect draws**, designed **without** multi-draw-indirect and **without** `indirect-first-instance` (browser limits); render bundles for stable content.
2. **Native** measurement — pin + record wgpu 29.x, GPU/driver, present mode (confirm the DXGI frame-pacing fix): GPU frame time + CPU submit, p50/p95/p99/max over ≥60 s including a worst-case "everything visible" frame, at 5k and 20k.
3. **Browser** measurement (Chrome/Edge + **Firefox 141**): same scene; confirm the largest storage buffer < 128 MB; confirm the culling path works without multi-draw-indirect.
</deliverables>

<success_criteria>
**Native:** 5k p99 ≤ **8.3 ms** (≥120 fps); 20k p99 ≤ **16.6 ms** (≥60 fps) with no frame-time spikes; draw-call count does **not** scale ~linearly with entity count (proves instancing engages). **Browser:** 5k p99 ≤ **16.6 ms** (≥60 fps) Chrome/Edge; 20k **≥ 30 fps acceptable** for the lite-editor (gap from native explained by draw-call count, not a correctness bug); runs on Firefox 141. Numbers quoted; verdict recorded.
</success_criteria>

<verification>
Two runs per discipline (median + p99); investigate any >25% run-to-run swing (watch for the wgpu periodic-slow-frame pattern). Confirm draw-call count vs entity count (instancing sanity check — if it scales linearly, instancing isn't engaging). Browser: verify the storage-buffer ceiling and the no-multi-draw-indirect path explicitly.
</verification>
