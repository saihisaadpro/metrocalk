# Prompt 16 — M2.5: Editor-UI scaffold (`/editor`, React/TS)

> Use with `00-orchestrator.md` (v2). **Effort `xhigh`.** Builds the React/TS editor shell UI against the M2.4 transport contract. Independent of the shell-composition spike (the UI is the same whether composited via DComp or CEF) and of M2.2 (render).
> **Lane:** Owns `/editor` (React/TS — not a cargo member) · **Blocked by M2.4** (consumes the `DeltaTransport` TS client + frame contract) · Parallel-safe with M2.3 once M2.4 lands · see `prompts/README.md`.
> Read first: `metrocalk.md` (the intent-first north-star UX), ADR-006, invariants 1/2/4, the M2.4 protocol, and the research reality below.

---

<task>
Scaffold the editor UI as a **projection of the authoritative core**, not a second source of truth (invariant 1). Research-verified architecture (June 2026): the UI is a projection cache fed by deltas over the M2.4 transport — built on **`useSyncExternalStore`** (Zustand as the thin selector-backed wrapper) so a delta re-renders only the components whose slice changed (tldraw's proven model for canvas-scale scenes). The **schema-driven inspector** renders from the registry's JSON Schema via **JSON Forms** (renderer-registry + testers — the right fit for typed/semantic fields, vs RJSF's template model). The **binding graph** is React Flow, neighborhood-scoped + memoized. Edits apply **optimistically** then reconcile on the authoritative echo, and every rejected bind surfaces its reason ("every 'no' explained").
</task>

<scope>
In scope (`/editor`): the projection store (delta-fed, selector-subscribed); the TS `DeltaTransport` client (in-process-WASM for browser + a desktop binding that M2.6 will point at the real shell IPC); the schema-driven inspector (JSON Forms + custom renderers for the typed fields); the React Flow binding graph (neighborhood-scoped); the hierarchy/list panel (summary-projection + virtualized rows for 5k entities); optimistic-echo + reconciliation; the input-ownership contract (UI chrome = React; the viewport rect's hot input belongs to Rust — invariant 4 — events over the viewport are NOT handled in JS). Out of scope: the actual wgpu viewport + its composition (M2.3/M2.6), real assets, the AI/describe-to-create layer, the Rules UI (post-slice).
</scope>

<deliverables>
1. **Projection store** — `useSyncExternalStore`-backed (Zustand), keyed by entity id, updated immutably per-entity so reference equality changes only for touched entities; selector subscriptions (`useStore(s => s.entities[id].transform)`); a "summary projection" selector (`{id,name,parentId}`) for the list/hierarchy so the inspector's detail never re-renders the tree. Validate tear-free under a React 19 concurrent transition.
2. **Transport client** — TS side of `DeltaTransport`: decode M2.4 frames → apply to the projection store; encode edit transactions → send. Carry the `%EPH` ephemeral channel as a separate path (live preview/presence) from the `%LOR` committed-delta path.
3. **Schema-driven inspector** — JSON Forms with custom renderers registered via testers for Metrocalk's typed fields (`EntityRef` picker, bind-target selector ranked by the compat query, color/curve, enum); field `onChange` → optimistic edit + a JSON-Patch-shaped transaction (same shape the AI layer emits, so human/AI edits are one language). Use uncontrolled inputs inside renderers if per-keystroke re-render shows in profiling.
4. **Binding graph** — React Flow, **neighborhood-scoped** (show the selected entity's compatible/bound targets, expandable — never 5k nodes at once), strict `React.memo`/`useCallback`, cheap node CSS. Document Sigma.js as the 50k+ fallback (don't build it).
5. **Optimistic echo + reconciliation** — `displayed = authoritative_base ⊕ pending_local_ops`, client-op-id per edit, drop the pending op on its echo; **rejection is first-class UX** — a rejected bind removes the optimistic edge and shows the core's merge-validation reason (the north-star "every 'no' explained"). Keep reconciliation-undo separate from the engine-side user undo.
6. **Input-ownership contract (stub + documented)** — the React side claims only UI-chrome pixels; events over the viewport rect are left for the native layer (wired for real in M2.6). Document the hit-test boundary so M2.6 plugs in cleanly.
</deliverables>

<success_criteria>
A delta for entity X re-renders only X's subscribed components (verify with a render counter) — the 5k-entity hierarchy + a selected-entity inspector stay responsive; no tearing under concurrent transition; the inspector renders editable forms from a JSON Schema and a field edit produces a JSON-Patch transaction; the binding graph shows a neighborhood and stays smooth; an optimistic edit shows instantly and reconciles (no-op on confirm, visible correction + reason on reject); the input contract leaves viewport events for the native layer.
</success_criteria>

<verification>
Render-count instrumentation proving selective subscription at 5k entities (edit one entity → exactly its components re-render). Tear test under a forced concurrent transition. A reject path test (optimistic bind → core rejects → edge removed + reason shown). Profile the inspector for per-keystroke re-render. Adversarial paragraph: "the case React state quietly becomes a second source of truth" — and the guard against it.
</verification>
