# Prompt 06 — M1.2: ECS wrapper API + Flecs backend

> Use with `00-orchestrator.md` (v2) as system prompt. **Effort `max`** — this is the load-bearing M1 session: the engine's relational query surface is defined here, and a wrong abstraction is expensive to undo. Run **after M1.1**.
> Read first: ADR-001 (Flecs lives behind our trait, no leakage), **ADR-006** (two backends — native Flecs + a Phase-2 pure-Rust index over the Loro projection), the Flecs ergonomics + binding-safety notes in `spikes/flecs/README.md`, the frame-budget section of `M0-gate-review.md`, and invariants **4** (hot path never crosses JS) and **5** (every pre-1.0 dep behind our trait).

---

<task>
Define and implement the project-owned ECS query-API wrapper, with the Flecs backend behind it. This trait is the engine's relational query surface — both north-star UX tests run through it. Per ADR-006 a second backend (browser, pure-Rust over the Loro projection) must implement the SAME trait in Phase 2. Design the abstraction now so that second backend slots in without a rewrite, and so no `flecs_ecs` type ever leaks past the wrapper crate.
</task>

<scope>
In scope: the query-API trait; the Flecs backend implementing it; entity + relationship-pair create/remove/set; query build + iterate; read-target; deferred mutation; safety locks ON; the wrapper crate as the ONE place `unsafe` and `flecs_ecs` are permitted (the unsafe-forbid exception M1.1 set up).
Out of scope: the browser pure-Rust backend itself (Phase 2 — but the trait MUST admit it, proven by doc), the metadata registry (M1.3), the ECS↔Loro commit pipeline and undo (M1–2), UI, rendering. Do NOT build the second backend; design for it.
</scope>

<deliverables>
1. **Query-API trait — anchored to the PRODUCT's compatibility queries, not Flecs's feature set.** The contract is exactly the queries ADR-006 names: pair-match `(Provides, X)`, wildcard `(R, *)`, negation ("lacks `(BindsTo, *)`"), and read-target — plus the minimal entity/pair/component mutations they require. Not Flecs's full surface (the pure-Rust backend can't mirror it); not a lowest-common-denominator that strips native power. If a query the product needs isn't expressible, widen the contract deliberately and note it.
2. **Flecs backend** behind the trait — `flecs_ecs` types fully hidden. Resolve the spike's ergonomic sharp edge by exposing ONE clear form (e.g. the `with`/`without` value-vs-type ambiguity, and wildcard via the documented idiom) — pick one surface and document why. Deferred mutation; **safety locks ON** (the ~0–10% cost is the price of soundness per spike ②).
3. **Two-backend proof** — a crate-root doc comment mapping, concretely, what a pure-Rust-over-Loro backend implements for each trait method (per ADR-006), plus a mechanical check (compile test or CI `grep`) that no `flecs_ecs`/Flecs type appears in any trait signature or any crate outside the wrapper.
4. **Zero-cost-native check** — a benchmark showing the wrapper's cached compat query stays within run-to-run noise of the raw-Flecs spike number (12.2 µs p99 @5k). The abstraction must not tax the hot path (invariant 4). Report the wrapper-vs-raw delta explicitly.
</deliverables>

<success_criteria>
spike ②'s compatibility query expressed and passing through the wrapper trait · mechanical check finds zero Flecs types outside the wrapper crate · safety locks ON · wrapper compat query within ~noise of raw Flecs (12.2 µs p99 @5k) — quantified, not asserted · the two-backend doc comment concretely maps every trait method to a Loro-projection implementation (no method only Flecs could satisfy).
</success_criteria>
