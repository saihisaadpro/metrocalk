# Prompt 18 — M3.1: Binding-by-intent UX (the north-star, made real)

> Use with `00-orchestrator.md` (v2). **Effort `max`** — this is **north-star test #1**, the product's signature interaction and the reason Metrocalk exists. Serial; commit on `master`.
> **Blocked by M2.6** (it builds on the running editor: the inspector, the viewport, the real round-trip). Authored from the roadmap (plan §7, M3–5) + the *already-built* deterministic core — refine against M2.6's actual editor when you start.
> Read first: `metrocalk.md` (north-star #1) + `north-star-tests.md` (**test 1** — your acceptance script), ADR-010 (editor = projection), and the machinery that already exists: the M1.2 `World` compat query (pair-match / wildcard / **negation**), the M1.3 registry (`provides`/`requires`/`observes` + ranking surface), the M2.5 inspector + React Flow graph, the M2.6 round-trip.

---

<task>
Make north-star test #1 real: **click an entity → the engine reveals what it can connect to, ranked by intent, with every incompatible target's "why not" explained → one click binds → undo works → reload preserves.** ≤2 interactions, the compat query <16 ms (already proven at ~12 µs), every "no" explained. The deterministic machinery is *already built* — the registry's capabilities, the `World` compatibility query, the transactional bind. **This session is the UX that surfaces it** — the moment Metrocalk feels categorically easier than Unity/Unreal's inspector-and-wire workflow. It must work fully offline, no LLM (AI is a guest, not here).
</task>

<scope>
In scope: the **context-reveal** flow (select → run the existing compat query → rank → present); the **ranking** (proximity/recency/affinity, registry-fed, deterministic, offline); the **explainability** ("why not" for every incompatible target, from the registry's `requires`/`provides`); the **one-click bind** → JSON-Patch `EditTx` → commit pipeline → undoable; the visual binding affordance (highlight compatible targets in the viewport + the M2.5 React Flow graph). Out of scope: describe-to-create / AI composition (later M3+/Phase 2), the marketplace, generation, the Rules UI, real art assets.
</scope>

<deliverables>
1. **Context reveal.** On entity select, run the existing `World` compat query ("entities providing X not yet bound toward Y" — pair-match + negation) and get the candidate set. No new query engine — surface the one M1.2/M1.3 already proved.
2. **Ranking** — order candidates by **proximity** (scene distance), **recency** (last-touched), **affinity** (capability fit), deterministically and offline, fed by the registry. Document the ranking function (and write an ADR if its shape is a lasting choice).
3. **Explainability — "every no explained."** For every *incompatible* target, the reason, derived from the registry (missing required capability · already bound · type/dimension mismatch), surfaced on the greyed target (hover/inline). This is the north-star differentiator — make it genuinely helpful, not a generic "incompatible."
4. **One-click bind.** Click a candidate → the bind relationship as a JSON-Patch `EditTx` through the commit pipeline → reflected in the viewport highlight + the React Flow graph + the inspector → undoable (engine-side stack) → survives reload (Loro).
5. **The ≤2-interaction + latency proof** — instrument the interaction count (select + click = 2) and the compat-query+reveal latency on the real M1.4 5k scene.
</deliverables>

<success_criteria>
Selecting an entity surfaces **ranked** compatible targets within the frame budget (<16 ms query); a bind takes **≤2 interactions**; **every incompatible target shows its specific reason**; the bind round-trips, is undoable, and survives reload; the ranking is **deterministic** (same scene → same order). That is `north-star-tests.md` **test 1**, passing — run it as the literal acceptance script.
</success_criteria>

<verification>
Use `north-star-tests.md` test 1 as the acceptance checklist (each ☐). Measure compat-query + reveal latency on the 5k scene (twice, median + p99). **Dogfood it:** the adversarial paragraph is "the case the ranking feels wrong or a 'why not' is unhelpful" — actually try to bind things and judge whether it feels like the categorical win the vision promises; if it doesn't, that's the finding, not a pass.
</verification>
