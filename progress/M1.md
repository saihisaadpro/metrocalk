### 2026-06-13 (M1.6 — pipeline hardening: audit fixes before M2 builds on it)
An orchestrator audit of M1–2 surfaced four robustness gaps the editor (M3+) would hit — untested, so green CI hadn't caught them. Fixed each with a regression test that fails on the old code and passes on the new, before M2 depends on the pipeline.
- **Undo over-removal bug (real correctness defect) — fixed.** The inverse of an additive `SetField` (a field that didn't exist before) was `RemoveComponent`, which deleted the **whole** component — so undoing "add `Health.maxHp`" also destroyed `Health.hp` written by an **earlier** transaction. The in-tree comment called it "conservative but correct"; it was not correct across transactions. Added `Op::RemoveField { entity, component, field }` (removes one field, drops the component record if it empties — exact prior state) and made the additive-`SetField` inverse emit it. **Regression test** `undo_additive_set_field_preserves_sibling_field` + the new **per-single-undo property invariant** both **fail on the old `RemoveComponent` inverse and pass on the fix** (verified by a temporary revert: 2 failures → restore → green).
- **Atomic commit (claimed-but-unenforced) — enforced.** `commit` applied ops one-by-one with no rollback and didn't actually use `defer` despite the docstring. A mid-batch failure (a plugin/AI op referencing a stale entity — exactly what invariant 3 invites) left a partial mutation **and** pushed no undo entry → un-revertable. Now every op is **pre-validated** against a forward-simulated alive-set (existing + in-batch creates − in-batch deletes, incl. cascade) *before any apply*; a single bad op rejects the whole batch. Compute+apply is **interleaved** so an op may legally reference an entity created earlier in the same batch (create-then-set, create-then-delete). Docstring corrected (no false `defer` claim; the one non-atomic residual — a Loro-internal error mid-apply — is documented and kept loud). **Test** `failed_multi_op_transaction_is_atomic`: a 3-op tx with an invalid middle op ⇒ `Err`, zero state change, **no undo entry**; `intra_transaction_references_are_valid` covers the same-batch reference + create-then-delete cases.
- **O(1) `tid→eid` index.** `tid_to_eid` was an O(n) linear scan over `eid_to_tid`, called inside `collect_subtree`'s BFS and per-subtree-entity during delete-inverse capture → O(subtree × scene) on a large-subtree delete. Now a `tid → EntityId` map maintained alongside `eid_to_tid` (create/delete/rebuild). 
- **Loro errors propagated, not unwrapped, in `apply_*`.** Every `.unwrap()` on a fallible Loro op in the `apply_*` mutation path is now `?`→`PipelineError::Loro` (via a `try_child_map` get-or-create + `loro_err` helper). Undo/redo keep an `.expect()` **on purpose** (they replay an already-valid transaction; a failure there is a logic bug, surfaced loudly) — documented.
- **Re-audited the whole `InverseOp` set** for any other inverse coarser than its forward op: the additive-`SetField`→`RemoveComponent` case was the only one; all others (`DeleteEntity`→`ResurrectSubtree`, `RemoveComponent`→`RestoreComponent{fields}`, tag/pair/binding/reparent) are already exact.
- **Re-measured undo (twice, n=500 for a noise-stable p99 under `cargo test`'s parallel load):** SetField undo **median 0.07–0.11 ms / p99 0.24–0.30 ms**; entity-resurrection undo **median 0.31 ms / p99 0.72 ms** — all ≫ under the 5 ms budget (no regression; the hardening's cost is negligible). Bumped the bench sample 50→500 so p99 is a real percentile, not the single max (the old n=50 flaked when the bench shared the box with the rest of the suite).
- **49 core tests green** (was 44: +2 pipeline atomicity, +3 undo: additive-regression, RemoveField, per-single-undo property), workspace fmt + clippy `-D warnings` + both no-leak greps (`flecs_ecs`∉core/transport/plugins, `loro`∉transport/plugins/ecs) clean — all 4 CI gates.
- **Phase-2 carry-forward (recorded, not fixed here):** after `merge()`, `rebuild_ecs_from_loro` recreates entities but **not their ECS tags/pairs** — capabilities are ECS-only and never written to Loro, so the **compatibility query is empty after a merge**. The fix needs the registry (component-kind → capabilities) wired into the rebuild; scheduled with collab in Phase 2 (see `../progress.md` carry-forward).

### 2026-06-13 (M1.5 — 16 ms compatibility-query CI perf gate)
- **The product's interactivity promise (north-star test #1, <16 ms) is now a hard CI tripwire** — the 3rd, beside the `flecs_ecs` no-leak grep and the wasm build. `tools/query-gate` (a release bin; default lints, outside `/core`, like `scene-bench`) builds M1.4's shared **5k preset through the `World` wrapper** (reuses `build_scene`/`compat_clauses` — not a bespoke scene), runs the **cached** compat query 2000× after warm-up, and **exits non-zero if p99 > 16 ms** (with a `::error::` annotation). An integrity assert (`matched == 211`) refuses to time a broken query. `.github/workflows/perf-gate.yml` runs it on push + PR.
- **Calibration — an absolute product-budget gate, not `baseline × k`, not `criterion`.** Runner baseline (`ubuntu-latest` 2-vCPU, read from the gate's own `::notice::` on commit `b77587d`): **median 10.6 µs / p99 20.6 µs / max 35.4 µs → 776× under the 16 ms budget** (dev Windows desktop: 9.7 / 28.1 / 135 µs, ~570×). The runner is slower than the spikes' i9, yet the query is hundreds× under budget (it's `O(matches)`, and the compiled query is cached). Why the huge headroom is *right*, not lazy: at µs scale p99 on a shared VM is **relatively noisy** (a neighbour-VM hiccup swings the tail run-to-run) but **irrelevant against 16 ms** — a `baseline × 3` (~60 µs) gate would be *flaky*; the product-budget gate is rock-solid yet still catches the regressions that would actually break the promise (an accidental **uncached** path, an archetype-**fragmentation** blowup — *orders of magnitude*, not µs). `criterion` gates *relative* regressions vs a stored baseline (noise-sensitive, needs a committed baseline); we want an *absolute* frame-budget gate, so the gate is a small `exit(1)`-over-budget binary.
- **Verified green AND red on real CI** (spike ③'s wasm-tripwire discipline): **green** on every push (perf-gate success on `483215e`, `b77587d`, `8d28618`; two readable p99s **20.6 µs** and **21.8 µs** — ~6% apart, non-flaky); **red** on a throwaway branch (`verify/m1.5-perf-gate-red`, since deleted) where injecting `METROCALK_GATE_INJECT_SLOW_US=17000` drove p99 to **17 060 µs** → `::error:: … EXCEEDS the 16000 us budget` → exit 1 → **job failed**, while `matched` stayed 211 (the query is correct, just slowed). Only `.github/workflows/perf-gate.yml` differed on the branch; **`main` never carried the injection.**
- **Mechanics note:** unauth log download is 403 on the public repo, so the gate prints its measurement as a one-line `::notice::` — read back via the (unauth-readable) check-runs **annotations** API, so the calibration baseline above is a *measured* number, never hand-typed. Full write-up: `tools/query-gate/README.md`.
- **This closes M1's foundation arc (M1.1–M1.5).** Lane honored: M1.5 owns `.github/` + the query bench only; the parallel **M1–2** commit-pipeline work owns `/core` (untouched here). Next: M1–2 (`prompts/10-commit-pipeline.md`).

### 2026-06-13 (M1.4 — shared stress-scene generator + F1 re-measure)
- **One shared seeded generator** (`ecs::scene`): `SceneParams` (entities/caps/p_health/extra-caps/edges) + `preset_5k`/`preset_20k`, built **through the `World` trait only**, used by the ecs bench, core tests, AND `tools/scene-bench` — no duplication. Determinism via a backend-independent **FNV-1a digest** over generation decisions (not entity ids); pinned in a test (`DIGEST_5K=11506122691080605209`, `DIGEST_20K=1779473442952625601`) so **CI on ubuntu re-checks the same values = cross-OS byte-identical** proof. Dense and sparse variants share a digest (storage ≠ logical scene). 5k→211 compat / 1999 edges, 20k→830, all parity with spike ②.
- **F1 measurement** (`tools/scene-bench`, `memory-stats` RSS, one variant per process so RSS isn't contaminated; 2 runs each, median+p99 over 2000):

  | @20k | bytes/entity (RSS delta) | compat p99 @5k | compat p99 @20k | per-entity `targets()` |
  |---|---|---|---|---|
  | **dense** (fragmenting pairs) | **~8.8 KB** | 9–14 µs | 71–74 µs | ✅ works |
  | **sparse** (`DontFragment`) | **~2.4 KB** (3.6× less) | 71–79 µs | 245–278 µs | ❌ **broken** |

  (The spike's 14.8 KB/entity was peak-RSS/entity over the whole process; the cleaner *delta* baseline here is ~8.8 KB. DontFragment's 2.4 KB beats both.) Run-to-run swings <15% except the dense@5k p99 (9.2↔14.0 µs) — µs-scale tail jitter on a sub-millisecond measurement, medians stable; not a regression.
- **Verdict — M1 keeps DENSE storage.** DontFragment wins on memory (3.6×) but loses on functionality + latency: it **breaks per-entity `targets()`** (Flecs: "target_for does not yet work for DontFragment"; the compat query + `for_each_edge` still work, but the inspector's "what does this entity bind to?" read would silently break), and slows the query 3.5–8× (still ≪16 ms). The memory pressure doesn't bite at M1 scale (dense ≈176 MB @20k on native desktop), and the browser — where memory is tight — runs the **Loro** backend, not Flecs (ADR-006). `set_sparse` stays in the `World` trait as a *measured* lever to revisit for very-large native scenes (100k+) **once `targets()` is re-implemented via a sparse-compatible path**. This is the honest call: a real memory win that isn't worth its current costs yet. (Storage model unchanged → no architecture.md change.)

### 2026-06-13 (M1.3 — component metadata registry)
- **`metrocalk-core::registry` built — the data layer behind the compatibility query.** `Registry<W: World>` (generic over the M1.2 backend) interns each capability as an `Entity` and records every registered kind as an entity carrying `(Provides, cap)` / `(Requires, cap)` / `(Observes, cap)` pairs, so `providers_of("Health")` / `requirers_of` / `observers_of` answer **through the wrapper's pair-match query** — no reaching behind it, `flecs_ecs` stays out of `core` (CI grep stays green).
- **JSON-Schema registration + validation:** `ComponentMeta::builder(..).fields_from_json_str(schema)` parses the component-field subset (object/properties/scalar types/required/format/description→ui-hint) and rejects malformed schemas with **actionable, non-panicking** `RegistryError`s (root-not-object, unsupported-type naming the field, missing-type, required-unknown-field, invalid-JSON, …). Builder chosen over a derive macro (less magic; same runtime path a plugin/marketplace component uses).
- **Serialization round-trip:** `to_json`/`from_json`; fields canonicalized (sorted), `serde(skip)` on empties → all 12 stdlib components serialize → deserialize → re-serialize **byte-identical**. `stdlib::standard_components()` = 12 real kinds (Transform, Health, HealthBar, Sprite, MeshRenderer, RigidBody, Collider, AudioSource, Light, Camera, Animator, Script) with a real capability web (Sprite+MeshRenderer both provide `Renderable`; HealthBar requires+observes `Health`).
- Tests (6 green): capability queries through the wrapper (`providers_of("Renderable") == [MeshRenderer, Sprite]`, `requirers_of("Health") == [HealthBar]`), byte-identical round-trip per component, JSON-schema parse (types/required/format/hints), malformed-schema rejection (6 cases, each actionable), empty-name + duplicate rejection. Workspace clippy `-D warnings` + fmt clean.
