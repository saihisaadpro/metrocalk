## 2026-06-20 — M3.3: viewport context actions + hover details (the "context reveal")

M3.1 made binding discoverable through the side panel; M3.2 made it describable. M3.3 makes it
**direct**: right-click an entity in the viewport → a menu of exactly the actions valid for it, every
"no" explained; hover → a details tooltip you can read without selecting. Built split by durability —
the UI-agnostic substance in the bridge (survives the React port), a thin probe in the scaffold.
([ADR-016](../decisions/016-viewport-action-model-and-interaction.md).) Off the Phase-2 economy thread.

**What landed** (4 green increments on `master`/`origin/main`):

- **The durable core** (`0e9642e`): `editor-shell/src/actions.rs::actions_for(entity)` — the registry-
  driven action model returning valid actions + a **specific reason** for each unavailable one (the M3.1
  `why_not` discipline). `Bind…` is the only conditional action (available iff the entity has an unbound
  required cap — reuses the reveal's `required_caps`; greyed "requires no capabilities" / "already bound"
  otherwise); Remove/Duplicate/Focus/Inspect are always available for a live entity. Plus the
  transactional `capscene::{remove_entity, duplicate_entity}` — **Remove** is one undoable tx that deletes
  the entity *and* cleans every binding it's in (a removed provider's edge is freed → the dependent
  re-opens; removing a requirer clears the provider's consumed-marker `(BindsTo)` pair → it re-enters the
  candidate set), undo restores entity (M1.6 resurrection) + edges atomically; **Duplicate** clones
  components + provides/requires caps under a fresh deterministic id, BindsTo NOT cloned (independently
  bindable). `Record::Remove`/`Duplicate` → both survive reload (ADR-013). **5 headless tests.**
- **Shell commands** (`e0f670b`): `entity_actions` / `remove_entity` / `duplicate_entity` / `focus_entity`
  / `viewport_peek` (NON-mutating pick for hover) / `entity_details`, all through the engine thread + the
  one pipeline. `render.rs` gains a settable camera `cam_target` (Focus frames an entity; orbit/zoom stay
  relative; hot path unchanged). Remove sends a **targeted** delta (`Remove(id)` + `RemoveEdge` per freed
  binding — inv. 2, not a full reload); Duplicate echoes `project_entity(clone)`.
- **The live scaffold surface** (`1124160`): `web/index.html` — right-**click** disambiguation
  (a right-press that moved <6 px is a click → menu; a right-drag still orbits, zero-per-frame-IPC); the
  `#ctxmenu` populated by `entity_actions` (valid clickable, unavailable greyed with the reason); **hover**
  → a debounced (130 ms settle) `viewport_peek` + `entity_details` fetched **only on hovered-entity change**
  (never per frame, inv. 4) → a `#tooltip` (name · components · caps · tracking). e2e extended
  (right-click→menu→Remove→Ctrl-Z; hover→tooltip-without-selecting; right-drag→orbit-not-menu) + the stale
  M5 marketplace test corrected.
- **Latency** (`<this entry's commit>`; release, 2 runs, i9-13900H): `actions_for` on the 5k scene
  **p50 ~1.5–2.7 µs / p99 ~2.7–3.7 µs** — ≈4000× under the 16 ms budget, once per right-click, never per
  frame.

**Adversarial paragraph (the case it breaks):** right-click-during-orbit fires a spurious menu · hover
floods IPC at 60 Hz · removing a bound provider strands its dependent or undo fails to restore the edge ·
Duplicate aliases the original's id. **Guards:** the menu opens only on a <6 px right-press (a drag is an
orbit); hover is debounced + details fetched on-change-only (peek is non-mutating, 0 per-frame IPC);
Remove cleans the freed edge + the consumed-marker pair and undo restores the entity + edges atomically
(headless-proven); Duplicate allocates a fresh deterministic id (`alloc_entity_id`, never the source's).

**Honest boundaries:** the live shell is the minimal scaffold, not the React `/editor` — the production
menu/tooltip is the named **follow-up** (the durable action model + commands move over unchanged); the
live e2e runs against the packaged app (a local-GUI step, like the existing specs); the invariant-4
"0 IPC/frame during orbit" holds by construction (the right-drag path is unchanged) — the live `[viewport]`
capture is the local run. wasm parity by design (the action model is pure metadata, like the reveal; lives
in native bridge — full browser run awaits the Phase-2 backend); the `wasm32` tripwire is unaffected.

**Verification:** editor-shell suite green incl. the 5 action tests; `actions_for` latency measured ×2;
clippy `-D warnings` + fmt clean (lib, tests, `src-tauri` bin); JS syntax-checked. ADR-016 + index row;
`architecture.md` Binding-by-intent + Viewport-input rows extended. **Next:** the React `/editor` port of
the menu + tooltip.

## 2026-06-19 — Live reload: the bug was *surfacing*, not persistence (diagnosed by measurement)

A user reported binds not surviving close→reopen in the release shell (clicked HealthBar `1_1129`,
tracking live; after kill + relaunch, "no tracking"). **Diagnosed before fixing — measured, didn't
assume** (the prompt's assumed cause was refuted):

- **The persistence layer was never broken.** Replaying the user's *actual* on-disk log at the shell's
  real `SCENE_N=5000` restores `(applied 14, skipped 0)`; the live release exe prints `restored 14 edits
  (0 skipped)` and re-seeds to **5001** entities (5000 seed + the describe-created HealthBar). The
  `#mtk …` header matched `fingerprint(5000)` byte-for-byte; `project_full` already emitted every
  restored binding edge on connect. So the data round-trips correctly through the real path.
- **The real gap was UI surfacing on reload.** Nothing was auto-selected (panel showed its placeholder);
  the bound HealthBar's high id (`1_1129`) fell past the requirers list's first-N cap so it never showed;
  and the wgpu viewport drew **no** binding lines (it never did). The "tracking" was in the engine but
  invisible. (Plus the window reopened at the default position — a separate, real complaint.)

**Fix (minimal, at the cause):**
- **Panel** (`editor-shell/web/index.html`): requirers now sort **tracking-first** with a `tracking N`
  badge (id-order-independent), and `maybeRestoreFocus()` **auto-selects** the most-tracked requirer once
  on the first projection — so the restored binds show as "tracking ✓ …" with no hunting.
- **Viewport** (`render.rs` + `scene.wgsl`): a new wgpu `LineList` pass draws a **3D tracking line**
  between each bound entity's centres, built in `rebuild()` from `engine.bindings()`. It reuses the
  `Instance` storage layout as the point carrier (only `.center` is read by `vs_line`) and draws with an
  always-pass / no-depth-write state so a binding reads as an on-top overlay. Verified live: **275
  tracking-colour pixels** in the viewport region of a captured frame.
- **Window state** (`main.rs`): position+size saved to a sidecar `metrocalk-window.json` next to the exe
  by a 1 s write-on-change poll (robust to a hard terminal kill, which fires no close event), restored on
  setup. Verified live: window reopened at a seeded `300,140` then `120,80` (via `GetWindowRect`).

**Closed the test gap** (the part that makes it stay fixed): `editor-shell/tests/reload_surfacing.rs`
drives the live record stream (`Bind`/`Edit`/`Describe`/`Undo`) through a **real on-disk log at
`SCENE_N=5000`**, then re-seeds + replays (a true close→reopen) and asserts the net state (bind survives,
undone bind absent, edit restored to `2.5`, describe recreated at the same deterministic id) **and** that
`project_full` carries the surviving edge / not the undone one (the data→UI seam the panel + viewport
read). The live half is `editor-shell/e2e/specs-reload/reload.e2e.js` (+ `wdio.reload.conf.js`): launch
with a pre-seeded log → assert the real `.exe` surfaces the restored bind (auto-focus `restored …`,
`tracking` badge on `1_1129`, ≥2 boundrows with no click). **4/4 green.**

**Verification:** full `editor-shell` suite green (release; incl. the new test) · `clippy -D warnings`
+ `fmt` clean on both the lib and the `src-tauri` bin · reload E2E 4/4 live. The fingerprint guard +
id-determinism are intact (a genuinely incompatible-build log is still discarded — `persistence.rs`).
**Honest boundary:** the literal "kill + relaunch under one tauri-driver session" isn't expressible, so
the loop is split — headless proves write→replay restores; the reload E2E proves a restored log surfaces
in the live shell. Decision context: [ADR-013](../decisions/013-live-persistence-replay-log.md)
(dated status note appended). [ADR-002 not touched.]

## 2026-06-15 — M3.1 reveal engine landed (headless; live surfacing + dogfood pending)

**What landed.** The deterministic, offline **reveal → rank → explain** engine — north-star #1's
differentiator — built on the confirmed M1.5 `World` compat query + M1.3 registry, headless-tested and
latency-measured. `editor-shell/src/reveal.rs`, on `master` (`c7a1361`), green. ADR-011.

- **Reveal** (`reveal`): per the selection's required caps, the exact M1.5 indexed query
  (`with(Provides, C)` + `without(BindsTo, *)`) → compatible providers-not-yet-bound.
- **Rank** (ADR-011): proximity (Transform distance) → affinity (required-caps provided, via query-
  membership tally) → recency → stable entity-id tiebreak. Deterministic.
- **Explain every "no"** (`why_not`, O(1) per target, on demand): `MissingCapability("Health")` /
  `AlreadyBound` / `NoCapability` — specific, registry-derived, not generic.
- **One-click bind**: reuses the M2.6 `apply_edit` (Bind → `Op::AddBinding` → commit → undoable),
  already tested.

**Measured (release, 5k capability-bearing scene):** reveal (compat-query + rank) **p50 0.706 ms /
p99 1.107 ms** — well under the 16 ms budget. 4 tests: ranked-nearest-first · every-no-explained ·
deterministic · the 5k latency proof. clippy `-D warnings` + fmt clean.

**Honest finding (kept, not hidden):** two earlier drafts blew the budget at 5k — an eager
all-entities `why_not` scan (~33 ms) and a per-match `targets()` affinity read (~25 ms). The committed
design avoids both (indexed query + membership tally + on-demand `why_not`); that's why reveal is
scene-size-independent. Measured in **release** (debug was ~7–26 ms — not the metric, per the
benchmark discipline).
