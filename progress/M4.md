## 2026-06-19 — Entities render as real imported meshes (not cubes); describe-to-create is visually true locally

The describe-to-create promise was "a working, **visible** object", but every entity rendered as an
M2.2 placeholder cube — there was no real art and no asset path. M4 builds the local **mechanism**: a
real glTF/glb imports through a trait, becomes an internal mesh + a content-addressed handle, and an
entity carrying that handle renders **as that mesh** in the live viewport — undoably, persisted,
`wasm32`-portable. ([ADR-014](../decisions/014-asset-model-and-import-pipeline.md).)

**What landed** (4 green increments on `master`/`origin/main`):

- **New `metrocalk-assets` crate** (`df00ae0`) — the asset substrate, fenced off from `/core` (Flecs),
  Loro, and C-FFI so it compiles to `wasm32-unknown-unknown` (ADR-006, deliverable 6), exactly the fence
  that keeps Flecs out of the browser build:
  - `mesh.rs` internal `MeshAsset` (positions/normals/uvs/indices + materials + RGBA8 textures + bounds)
    — no foreign decoder types (invariant 5).
  - `source.rs` `MeshSource`/`AssetImporter` trait + `ImportError`, with the **adversarial guards**:
    64 MiB / 8M-element caps, fail-fast on a hostile glTF's declared accessor counts *before* allocating.
  - `gltf_import.rs` — the ONE file touching `gltf::`/`image::` (CI grep-gated, like `flecs_ecs`-in-
    `/ecs`); `Gltf::from_slice` on the glb blob + PNG texture decode from bufferViews. `gltf`
    `default-features=false` deliberately avoids the rayon'd decoder path that breaks wasm.
  - `store.rs` `AssetStore`: id-keyed, **content-addressed** (FNV-1a 128, zero-dep), beside the doc —
    geometry never enters ECS/Loro (invariants 1 & 2); idempotent import → the same handle re-resolves
    after reload (ADR-013 id determinism).
  - `gpu.rs` `MeshGpu` packing (merge primitives, bake material color to vertices, derive smooth normals)
    — the `wasm32`-portable render-data prep.
  - `demo.rs` dependency-free `.glb` encoder + own demo meshes (framed health-bar, octahedron prop,
    textured quad); `examples/gen_fixtures.rs` writes the checked-in `editor-shell/assets/*.glb`.
  - **8 unit tests** (import geometry/bounds/materials/texture, derive-normals, content-address
    stability, store idempotence, oversize/malformed guards).

- **Asset handle wired through the ECS + render** (`1df3b2d`): `capscene::place_mesh` (one undoable
  commit: Transform + `MeshRenderer.mesh` handle + `Renderable`); `instantiate`/`describe_create` gain a
  `MeshCatalog` (kind→handle) so a resolved kind with an asset instantiates carrying its handle, and a
  kind without one honestly falls back to the cube; `persist::Record::PlaceMesh` + the catalog on replay
  → a *visible* described/placed object survives close→reopen. `render.rs`+`scene.wgsl` add a mesh pass:
  per-asset vertex/index buffers drawn **instanced** across the entities using each asset (non-bindless,
  ADR-003), built/uploaded on the render thread (hot path off JS, inv. 4); the cube path stays as the
  placeholder/fallback + perf baseline; entities partition by mesh slot each revision. `main.rs` imports
  the embedded demo `.glb` once at startup (the one-shot heavy op), builds the catalog + slot/scale
  tables, and routes each entity's `MeshRenderer.mesh` → slot.
  - **Headless acceptance** (`editor-shell/tests/mesh_assets.rs`, 6): import a checked-in glTF asserts
    internal geometry/bounds → `place_mesh` is one undoable tx carrying only the handle → placed mesh
    survives export→replay (handle re-resolves) → describe-create attaches the catalog mesh or honestly
    falls back → an unknown handle still commits (renderer cube-fallbacks). Full `editor-shell` suite
    **28 green in release** (incl. the release-timing gates).

- **CI gates** (`4f961b9`): `wasm-tripwire.yml` gains a `wasm32-assets` job (`-p metrocalk-assets` for
  wasm32 — pulls neither core nor ecs, so Flecs is never built for wasm; **verified green on CI**).
  `ci.yml` gains a grep gate failing the build if `gltf::`/`image::` leak outside `gltf_import.rs`. The
  demo checker-PNG is now a byte literal so `image::` stays confined to the importer wrapper.

- **Measurements** (`0df0f4b`; release, ≥2 runs, **i9-13900H / RTX 4060 / Iris Xe, Win11 26200, rustc
  1.92.0; gltf 1.4.1, image 0.25.10**):
  - **Import (one-shot, never extrapolated):** `healthbar.glb` (2648 B, 48 verts) warm p50 ~21 µs / p99
    ~31 µs (cold ~377–630 µs); `prop.glb` ~10 µs; GPU pack ~2 µs. A sub-ms load-time op.
  - **Frame budget on a real-mesh scene:** an offscreen GPU bench (`mesh_frame_bench` example, the *same*
    pipelines + WGSL as the live viewport) on **5000 cubes + 200 instanced meshes** → CPU-submit p50
    ~0.09 ms / p99 ~0.25 ms; **CPU+GPU(wait) p50 ~0.21 ms / p99 ~0.40–0.52 ms** over 2 runs — **≈30–40×
    under the 16 ms budget** on the RTX 4060 (Vulkan). Pixel readback proves the imported meshes draw
    (red health-bars + teal octahedra). **Evidence:** `editor-shell/evidence/m4-mesh-scene.png`.

**Honest boundaries (named, not stubbed):** KTX2/basis-universal GPU-texture compression is **native-only**
(C++ FFI — the documented native/browser divergence; the importer decodes textures to RGBA8 and the
render bakes material base-color, with in-shader texture sampling the next render increment); collider
generation, LODs, and rig detection are physics/runtime-tier; base64/external-buffer `.gltf` import and a
UI import affordance are follow-ups. The **live in-window screenshot + dogfood "feels real" verdict** stay
human-gated (the offscreen bench is the automatable companion to the live `[viewport]` instrumentation).

**Adversarial paragraph (the case it breaks):** a malformed or pathologically-large glTF, or geometry
leaking into the scene doc and blowing up delta size / breaking reload. **Guards:** import validates +
size/element-caps (fail-fast on declared counts before allocating) and a malformed asset is rejected at
the trait, never reaching the doc; the doc/ECS carry **only the handle** (invariant 2); the store is
separate and content-addressed; the importer's `NoGeometry`/`Malformed`/`TooLarge`/`TooManyElements`
paths are unit-tested. An unknown handle is a *visible* cube fallback, not a crash.
